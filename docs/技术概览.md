# 技术概览

## 1. 项目定位

本项目是一个面向出版社场景的 AI 助手系统，当前以“编辑 / 教辅运营”作为核心用户，围绕以下能力构建：

- 登录鉴权
- 本地文档知识导入
- 基于 RAG 的智能问答
- 拍照搜题 / OCR 识别
- 对话历史记录
- 管理后台式 Web 前端

当前项目强调三点：

- 可运行：本地环境可直接启动、导入知识、发起问答
- 可理解：模块边界清晰，核心代码尽量保持直接
- 可迭代：预留模型提供方切换、多模态扩展和后续多 Agent 编排空间

## 2. 当前技术栈

### 后端

- Python 3.11+
- FastAPI
- LangChain
- SQLAlchemy
- FAISS
- Pydantic Settings
- Typer

### 前端

- React

### 当前已支持的模型提供方

项目已经从“只支持 OpenAI”扩展为“支持多提供方配置切换”：

- OpenAI
- ARK / 豆包（OpenAI 兼容）
- GLM / 智谱（OpenAI 兼容）

通过 `.env` 中的 `LLM_PROVIDER` 切换：

- `openai`
- `ark`
- `glm`

## 3. 当前后端核心模块

### 3.1 Provider 层

路径：`backend/app/services/providers.py`

职责：

- 根据 `LLM_PROVIDER` 选择具体模型提供方
- 为文本问答提供 `llm`
- 为图片 OCR 提供 `vision_llm`
- 为 RAG 向量检索提供 `embeddings`

当前设计特点：

- 文本模型、视觉模型、向量模型分开配置
- 视觉模型支持回退到文本模型
- 对上层业务保持统一接口，业务代码不关心底层是哪家模型

### 3.2 配置层

路径：`backend/app/core/config.py`

职责：

- 从仓库根目录 `.env` 读取配置
- 管理数据库、JWT、模型、向量库、上传目录等配置
- 对 `BACKEND_CORS_ORIGINS` 做兼容解析

### 3.3 RAG 层

相关路径：

- `backend/app/knowledge/loaders/file_loader.py`
- `backend/app/knowledge/splitter.py`
- `backend/app/knowledge/vector_store.py`
- `backend/app/services/knowledge.py`

职责：

- 解析 TXT / Markdown / PDF / DOCX
- 切片
- 向量化
- 写入本地 FAISS 索引
- 检索时按角色、书名、文档类型过滤

本次调试补充：

- 文本文件加载改为 `utf-8-sig`
- 这样可以自动去掉 BOM，避免知识内容前缀脏字符污染回答与终端输出

### 3.4 问答与 OCR 层

相关路径：

- `backend/app/services/chat.py`
- `backend/app/services/image_ocr.py`
- `backend/app/agents/orchestration/service.py`

职责：

- 文本问答
- 图片 OCR 识别
- 识别后继续走现有问答链路
- 将问答结果和来源保存到历史记录

当前链路：

- 文本问答：`llm + embeddings`
- 图片问答：`vision_llm + llm + embeddings`

### 3.5 鉴权层

相关路径：

- `backend/app/auth/service.py`
- `backend/app/auth/dependencies.py`

职责：

- 登录
- JWT 校验
- 管理员权限校验

当前已统一返回中文错误信息，便于前端直接展示。

## 6. 当前已知问题

### 6.1 Windows 终端中文输出容易受编码影响

这不影响浏览器前端和接口真实返回值，但会影响 PowerShell 直接打印中文时的显示效果。

### 6.2 JWT 默认密钥仍偏弱

当前 `.env` 里的 `JWT_SECRET_KEY` 仅适合本地开发，正式使用前应更换为更长的随机字符串。

### 6.3 FastAPI 使用了 `on_event`

当前仍可运行，但后续建议迁移到 lifespan 写法，避免 deprecation warning。

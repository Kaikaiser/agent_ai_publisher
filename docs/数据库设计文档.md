# 数据库设计文档

## 1. 说明

本文档描述“出版社 AI 助手”当前后端数据库设计。

当前数据库类型：

- SQLite

默认数据库文件由 `.env` 中的 `DATABASE_URL` 指定。

当前默认值：

```text
sqlite:///./data/app.db
```

## 2. 设计目标

当前数据库设计主要服务于四类业务：

- 用户认证
- 知识导入记录
- 文档元数据管理
- 问答历史保存

V1 重点是：

- 结构足够简单
- 本地运行方便
- 后续可迁移到 PostgreSQL

## 3. 表设计总览

当前主要有 4 张表：

- `users`
- `documents`
- `ingestion_jobs`
- `conversations`

## 4. users 表

用途：

- 保存登录用户
- 保存角色与密码哈希

字段：

- `id`
  - 主键
  - 自增整数
- `username`
  - 用户名
  - 唯一索引
- `password_hash`
  - 密码哈希
- `role`
  - 当前角色
  - 例如：`admin`、`user`
- `is_active`
  - 是否启用
- `created_at`
  - 创建时间

设计说明：

- 当前版本不拆用户资料表
- 角色直接存在用户表里，适合首版
- 后续若升级到 RBAC，可新增角色表和权限表

## 5. documents 表

用途：

- 保存已导入文档的元数据
- 不直接保存全文内容

字段：

- `id`
  - 主键
- `filename`
  - 原始文件名
- `file_path`
  - 本地保存路径
- `book_title`
  - 书名
- `doc_type`
  - 文档类型
- `allowed_role`
  - 允许访问的角色
- `source_type`
  - 数据来源类型，当前默认 `upload`
- `created_by`
  - 创建人用户名
- `created_at`
  - 创建时间

设计说明：

- 文档全文当前不进 SQLite，而是进入 FAISS 相关索引流程
- SQLite 主要保存文档业务元数据
- `allowed_role` 用于检索过滤

## 6. ingestion_jobs 表

用途：

- 记录导入任务执行状态
- 方便后续扩展为异步导入或任务面板

字段：

- `id`
  - 主键
- `filename`
  - 文件名
- `status`
  - 状态，例如：`pending`、`processing`、`completed`、`failed`
- `message`
  - 说明信息
- `created_by`
  - 执行导入的用户
- `created_at`
  - 创建时间

设计说明：

- 当前导入流程还是同步执行
- 但保留导入任务表，可以为后续异步化留接口

## 7. conversations 表

用途：

- 保存问答历史

字段：

- `id`
  - 主键
- `username`
  - 发起问答的用户名
- `question`
  - 用户问题
- `answer`
  - 系统回答
- `grounded`
  - 是否基于知识依据生成
- `sources_json`
  - 引用来源 JSON 字符串
- `created_at`
  - 创建时间

设计说明：

- 当前没有单独拆来源表
- `sources_json` 直接存 JSON 字符串，便于快速落地
- 如果后续需要做来源级分析，可拆分成独立表

## 8. 当前关系说明

当前数据库关系比较轻量，没有显式定义复杂外键。

业务上隐含关系如下：

- `documents.created_by` 对应导入人
- `ingestion_jobs.created_by` 对应导入人
- `conversations.username` 对应发起问答的用户

为什么当前没做复杂关联：

- 首版目标是快速稳定落地
- SQLite 本地单机运行优先
- 复杂外键关系对当前收益不高

## 9. 数据流与数据库协作方式

### 9.1 登录

- 从 `users` 表读用户信息
- 校验密码哈希
- 不落额外会话表，直接发 JWT

### 9.2 导入文档

- 写 `ingestion_jobs`
- 写 `documents`
- 文本切片与向量索引进入 FAISS，不直接进 SQLite 全文表

### 9.3 问答

- 不直接读 `documents` 表做全文检索
- 检索走 FAISS
- 最终结果写入 `conversations`

### 9.4 历史查询

- 从 `conversations` 表按用户名查询

## 10. 为什么当前不用 PostgreSQL

原因：

- V1 重点是本地快速启动
- SQLite 零部署成本低
- 足够承载当前用户、任务和历史数据

局限：

- 不适合复杂并发
- 不适合多用户生产环境
- 后续扩展能力不如 PostgreSQL

所以当前的设计是：

- 先用 SQLite
- 后续平滑迁移 PostgreSQL

## 11. 为什么全文不存数据库，而交给 FAISS

因为当前问答的主检索方式是向量检索，而不是数据库全文检索。

设计取舍：

- 数据库保存业务元数据
- FAISS 保存向量索引与检索能力

这样分工的好处：

- 查询路径更清晰
- 向量检索能力独立
- 后续更容易把向量层替换成云向量库

## 12. 当前数据库设计的优点

- 简单，适合本地开发
- 结构容易理解
- 与当前业务闭环直接对应
- 为后续扩展预留了空间

## 13. 当前数据库设计的限制

- 没有复杂外键关系
- 没有审计字段如更新时间、删除标记
- 没有用户管理日志
- 没有细粒度权限表
- 没有导入切片明细表

## 14. 后续数据库演进建议

### 14.1 短期建议

- 增加 `updated_at`
- 增加普通用户创建能力
- 增加文档更多元数据字段

### 14.2 中期建议

- 从 SQLite 迁移到 PostgreSQL
- 拆分来源表或消息表
- 增加用户操作日志
- 增加角色权限表

### 14.3 长期建议

- 引入多租户数据隔离
- 引入文档版本管理
- 引入导入批次与异步任务体系

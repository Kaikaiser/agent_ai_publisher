# 人工智能出版助手决策检索记忆技术文档

## 1. 文档目的

本文档用于沉淀当前项目在以下三个方向上的技术设计与落地结果：

- 检索增强生成优化
- 记忆体系优化
- 受控自主决策优化

同时给出后续可继续推进的优化方向，并回答是否需要引入 `MCP`。

本文档面向产品、后端、前端和后续维护人员，重点强调当前方案为什么这样设计、已经做到什么程度、下一步还能往哪里演进。

## 2. 当前项目定位

该项目不是通用聊天机器人，而是一个面向出版社与读者场景的书级智能助手平台。核心约束有三条：

1. 必须以当前书为边界运行，不能串书。
2. 回答必须尽量基于当前书知识，不允许大范围自由发挥。
3. 系统既要有一定自主决策能力，又必须可控、可审计、可解释。

因此，本项目最终采用的是：

- 书级隔离的检索与生成架构
- 分层记忆架构
- 受控的类 ReAct 决策执行链

而不是开放式、无限制的通用智能体架构。

## 3. 总体架构概览

当前后端已经形成四层协同结构：

### 3.1 知识层

负责书籍内容的事实检索。

- 长期存储：`PostgreSQL + pgvector`
- 关键词检索：`Elasticsearch`
- 检索策略：稠密检索 + 词法检索 + 融合排序 + 可选重排

### 3.2 记忆层

负责会话上下文、书级规则、项目规则和用户偏好。

- 会话记忆：`Redis`
- 长期记忆：`PostgreSQL + pgvector`
- 记忆作用域：
  - `session`
  - `book`
  - `project`
  - `user`

### 3.3 决策层

负责：

- 识别问题意图
- 选择执行路径
- 控制工具顺序
- 根据观察结果决定继续检索、澄清、保守回答或结束

当前实现为受控的两步决策闭环，而非开放式多步智能体。

### 3.4 生成层

负责把以下上下文合并后交给模型生成最终回答：

- 决策策略
- 执行轨迹
- 会话与长期记忆
- 当前书检索结果

生成层始终遵循：

- 书中事实以检索结果为准
- 记忆只用于风格、偏好、上下文和规则

## 4. 检索增强生成优化

## 4.1 原则

本项目的检索优化目标不是单纯提高召回量，而是同时满足：

- 当前书范围内检索
- 更高的命中质量
- 更低的串书风险
- 更稳定的问答表现

## 4.2 当前实现

当前检索链已经从旧方案切换为：

- `PostgreSQL + pgvector` 负责向量存储
- `Elasticsearch` 负责词法检索
- 融合策略采用倒数排序融合
- 可选接入外部重排服务

已落地能力包括：

1. 书籍文档重新导入与重建索引
2. 稠密检索与词法检索融合
3. 结果重排支持自动降级
4. 命中结果带来源和位置信息
5. 检索过程与决策过程可记录日志

## 4.3 本轮新增的检索控制能力

为了配合自主决策链，本轮额外增强了两点：

### 4.3.1 受控重试

第一次知识检索没有命中时，系统不会立刻结束，而会：

1. 先判断是否允许重试
2. 对查询做一次本地归一化
3. 仅在当前书范围内做一次重试

该重试具备以下约束：

- 最多一次
- 不调用外部模型
- 不跨书
- 不改变书级过滤条件

### 4.3.2 证据质量判断

系统不再只看“有没有命中”，还会对命中结果做轻量质量评估：

- `none`
- `weak`
- `medium`
- `strong`

判定依据包括：

- 命中文档数量
- 命中文本长度
- 命中来源分散度

该信号会直接影响后续决策：

- 弱证据时优先澄清或保守回答
- 中强证据时允许继续正常回答

## 5. 记忆体系优化

## 5.1 为什么不能只保存对话历史

原始对话历史只是日志，不适合直接参与后续问答。原因包括：

- 噪音大
- 结构弱
- 检索效果差
- 容易污染跨轮回答

因此项目采用的是“对话记录”和“可检索记忆”分离的设计。

## 5.2 当前分层记忆模型

### 5.2.1 会话记忆

作用：

- 维护当前会话短期上下文
- 维持追问连续性
- 支撑澄清恢复

存储：

- `Redis`

特点：

- 读取快
- 支持过期
- 适合短期上下文

### 5.2.2 书级记忆

作用：

- 保存某一本书自己的长期互动规则
- 保存书级人格、口吻、限制、书级上下文

存储：

- `PostgreSQL + pgvector`

### 5.2.3 项目记忆

作用：

- 保存多个书共享的规则
- 例如系列术语、产品线口径、项目规范

存储：

- `PostgreSQL + pgvector`

### 5.2.4 用户记忆

作用：

- 保存登录用户长期偏好
- 保存用户常见任务和回答风格偏好

存储：

- `PostgreSQL + pgvector`

## 5.3 当前记忆写入策略

每轮对话完成后，系统不会把整段对话直接长期保存，而是做候选抽取：

- 会话上下文写入 `session memory`
- 书级规则写入 `book memory`
- 项目级规则写入 `project memory`
- 用户偏好和任务写入 `user memory`

澄清轮次不会写入长期记忆，避免噪音污染。

## 5.4 当前记忆读取策略

每次问答时，系统会按以下顺序取记忆：

1. `session`
2. `book`
3. `project`
4. `user`

但事实优先级始终低于知识检索结果。也就是说：

- 检索结果负责事实
- 记忆负责上下文、偏好、规则和风格

## 6. 受控自主决策设计

## 6.1 设计目标

项目需要具备自主决策能力，但不适合做成完全开放式智能体。原因包括：

- 书级边界严格
- 回答准确性要求高
- 需要可审计
- 需要前后台可解释

因此当前采用的是：

- 受控决策
- 有限动作
- 有限步数
- 明确日志

## 6.2 当前执行链

当前系统已经具备一个受控的类 ReAct 执行链：

1. `Thought`
2. `Action`
3. `Observation`
4. 再次判断
5. 最终回答、澄清或保守退出

## 6.3 当前可执行动作

动作被限制在白名单内：

- `knowledge_retrieval`
- `retry_knowledge_retrieval`
- `memory_retrieval`
- `calculate`
- `clarify_user`
- `final_answer`
- `refuse_answer`

这意味着系统有决策能力，但不会演变成不受控的工具调用系统。

## 6.4 当前自主决策能力

当前系统已具备以下能力：

1. 意图识别
2. 基于书籍模式选择路由
3. 一次观察后的二次决策
4. 一次受控检索重试
5. 结构化澄清
6. 弱证据时保守处理
7. 全链路执行轨迹记录

## 6.5 当前不是完整开放式智能体

当前方案仍然有明确边界：

- 最多两步
- 工具固定
- 不自发扩展外部资源
- 不跨书
- 不做长时间挂起执行

因此它更准确的定位是：

**受控自主决策框架**

而不是：

**开放式通用智能体**

## 7. 澄清与恢复机制

本轮优化中，澄清不再只是返回一句提示文案，而是形成了可恢复状态。

## 7.1 结构化澄清

系统会给澄清返回一个槽位，例如：

- `topic`
- `chapter_or_section`
- `chapter_or_topic`
- `character_or_plot`

同时返回对应澄清文案。

## 7.2 会话持久化

待澄清状态会写入当前会话，包括：

- 是否待澄清
- 原始问题
- 待补充槽位
- 澄清提示文案

## 7.3 恢复执行

用户在同一个会话继续输入补充信息时，系统会：

1. 读取待澄清状态
2. 合并原问题与补充信息
3. 重新规划决策
4. 成功后清空待澄清状态

这让澄清从“静态提示”升级成“可恢复执行”。

## 8. 当前方案为什么没有迁到状态图框架

当前项目没有迁到状态图框架，原因是：

1. 当前链路复杂度还在可控范围内
2. 核心问题仍然是业务规则和书级边界，而不是执行图能力不足
3. 当前两步受控闭环已经能满足阶段目标
4. 过早迁移只会增加复杂度和维护成本

因此，当前阶段最合理的策略是：

- 保留现有业务层
- 持续增强受控自主决策能力
- 只在真正需要更复杂状态图时再评估迁移

## 9. 未来还可以怎么优化

在当前决策、检索、记忆三条主链之外，这个项目还有几个很值得做的方向。

## 9.1 配置中心产品化

建议补齐 B 端配置能力，让编辑可以直接配置：

- 书籍模式
- 澄清策略
- 保守回答策略
- 引用策略
- 角色化开关
- 安全级别

这会让决策能力真正变成平台能力，而不只是代码能力。

## 9.2 决策日志可视化

建议在后台增加决策日志查看页，至少展示：

- 任务意图
- 路由名称
- 执行动作
- 证据质量
- 是否澄清
- 是否重试

这对调试和运营都很重要。

## 9.3 数据治理

建议加强：

- 文档版本控制
- 重复导入检测
- 项目规则审核
- 记忆归档与人工修正

## 9.4 质量评估体系

当前已有离线评测能力，后续可以继续补：

- 检索召回评测
- 引用准确率评测
- 弱证据误答率评测
- 各类书籍模式的答复质量对比

## 9.5 前端体验

建议前端消费以下信号：

- `clarification_slot`
- `evidence_quality`
- `route_name`
- `decision_mode`

这会显著提升用户对系统行为的理解。

## 9.6 工程基础

建议继续收口：

- 数据库迁移体系
- 生命周期管理
- 配置分层
- 全链路结构化日志

## 10. 是否需要引入 MCP

## 10.1 当前结论

当前项目**不是必须立即引入 `MCP`**。

原因很直接：

1. 当前工具集规模不大
2. 当前工具边界很清晰
3. 当前最重要的是书级隔离和可控决策
4. 这些问题靠现有业务层就能解决

也就是说，当前项目的主要瓶颈不在“工具协议不统一”，而在“产品策略、检索质量、记忆治理和后台能力”。

## 10.2 什么情况下值得引入 MCP

当后续出现以下需求时，可以认真评估：

1. 需要把工具能力开放给多个执行器复用
2. 工具数量明显增多
3. 后端希望统一工具注册、调用和审计方式
4. 后续需要接更多外部工具
5. 希望把检索、规则查询、评测、图像能力统一成标准工具

## 10.3 如果引入，应该怎么引

如果后续引入，建议遵循一个原则：

**先引工具协议，不改业务边界。**

也就是说：

- 可以用 `MCP` 统一工具调用方式
- 但不能把书级隔离交给工具调用方自己决定

正确做法应该是：

- 工具仍然是书级作用域工具
- `book_id` 在工具实现层锁定
- 不暴露全局跨书检索能力

例如：

- 可以有“当前书知识检索工具”
- 不应该有“任意传入书编号的全局检索工具”

## 10.4 当前建议

当前阶段建议：

- 先不把 `MCP` 当成核心任务
- 等工具体系更复杂时再引
- 即使引入，也只用于标准化工具层，不替代业务策略层和书级边界控制

## 11. 结论

当前项目已经从传统的“固定检索后直接回答”升级为：

- 具备混合检索能力的书级知识系统
- 具备四层记忆能力的上下文系统
- 具备受控自主决策能力的类 ReAct 执行系统

当前方案的价值不在于追求最复杂的智能体框架，而在于：

- 可控
- 可解释
- 不串书
- 可继续演进

在这个前提下，后续更值得投入的方向是：

- 配置中心产品化
- 日志可视化
- 质量评估
- 数据治理

而不是过早引入更重的执行框架。

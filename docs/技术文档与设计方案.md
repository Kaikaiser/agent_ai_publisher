# 出版社 AI 助手技术文档与设计方案

## 1. 项目背景

“出版社 AI 助手”是一个面向出版社、教辅、内容运营与知识服务场景的 AI 系统。项目目标不是做一个单点 Demo，而是构建一个可运行、可理解、可迭代的 Agent 工程底座。

当前 V1 聚焦以下业务价值：

- 让管理员可以导入图书、教材、教辅资料
- 让登录用户可以围绕知识库进行问答
- 让系统具备后续扩展到图片、语音、多 Agent、更多出版业务工具的能力

## 2. 设计目标

### 2.1 核心目标

- 可运行：本地能启动、能登录、能导入、能问答、能查历史
- 可理解：目录清晰、模块职责明确、适合初学者阅读
- 可迭代：后续能扩展到多模态、更多工具、更多角色与更强 RAG

### 2.2 非目标

V1 暂不追求：

- 复杂多 Agent 协作执行
- 生产级高并发部署方案
- 复杂 RBAC 与租户隔离
- 完整 OCR、拍照搜题、语音识别链路
- 长期记忆自动注入

## 3. 总体方案

项目采用单仓库结构：

```text
backend/   Python + FastAPI + LangChain + SQLAlchemy + FAISS
frontend/  React + Vite
docs/      技术文档与运行说明
```

整体采用“前后端分离 + 后端分层”的设计：

- 前端负责登录、问答、知识导入、历史查看
- 后端负责认证、权限、文档导入、RAG 检索、Agent 问答、历史保存
- LangChain 作为模型编排和提示词调用的核心框架
- SQLite 作为本地业务数据存储
- FAISS 作为本地向量检索底座

## 5. 核心功能设计

### 5.1 登录与权限

当前角色模型：

- `admin`
- `user`

权限边界：

- `admin`
  - 登录
  - 知识导入
  - 问答
  - 历史查询
- `user`
  - 登录
  - 问答
  - 历史查询
  - 不能导入知识

设计取舍：

- V1 不做复杂 RBAC，避免过早引入权限矩阵复杂度
- 先保证管理员与普通用户的主流程明确
- 后续如需编辑、运营、教师等业务角色，可在此基础上升级

### 5.2 知识导入

支持的文件类型：

- TXT
- Markdown
- PDF
- DOCX

导入流程：

1. 管理员上传文件
2. 文件保存到本地上传目录
3. 根据文件后缀选择解析器
4. 提取文本内容与基础元数据
5. 进入切片流程
6. 生成向量并写入 FAISS
7. 将文档记录和导入任务写入数据库

导入记录包含：

- 文件名
- 书名
- 文档类型
- 允许访问角色
- 创建人
- 导入状态

### 5.3 RAG 设计

RAG 是 V1 的主链路，而不是附属功能。

#### 数据来源

当前知识来源是本地文件导入。

这样设计的原因：

- 最容易从 0 启动
- 最适合本地开发和演示
- 后续可以平滑替换成 CMS、数据库或对象存储接入

#### 切片策略

采用“平衡型分层切片”：

- 优先保留文档天然结构
- 再按统一长度做切分

当前参数：

- `chunk_size = 700`
- `chunk_overlap = 100`

设计原因：

- 过大的块会降低检索精度
- 过小的块会让知识过于碎片化
- 700/100 是首版兼顾稳定性和效果的折中

#### 检索策略

当前使用：

- OpenAI Embeddings
- FAISS 本地向量检索
- Top-K 相似度召回
- 元数据过滤

支持的过滤维度：

- `book_title`
- `doc_type`
- `allowed_role`

#### 来源引用

回答结果返回来源信息：

- `filename`
- `book_title`
- `doc_type`
- `location`
- `content`

这样做是因为出版场景更重视：

- 可追溯
- 可验证
- 可引用

### 5.4 Agent 设计

#### 当前实现策略

V1 采用单 Agent 主链路，保留多 Agent 扩展接口。

当前 `AgentOrchestrator` 负责：

- 触发知识检索
- 判断是否有知识依据
- 在有上下文时调用模型生成答案
- 在无依据时返回保守回答
- 在纯算式场景调用计算器逻辑

#### 为什么这样设计

原因有三个：

1. 先保证主链路可运行
2. 避免在 V1 引入复杂编排成本
3. 仍保留向多 Agent 扩展的模块边界

#### 当前工具

- 知识库检索工具
- 计算器工具

工具设计目的：

- 知识库检索工具负责出版场景主能力
- 计算器工具用于验证“工具调用能力”和简单确定性计算场景

### 5.5 问答历史

问答历史当前做两件事：

- 保存
- 查询

当前不做：

- 自动上下文拼接
- 长期记忆注入

原因：

- 先把数据链路跑通
- 避免首版把上下文管理复杂化
- 后续可在 `ConversationService` 和 Agent 编排层上演进

### 7.1 认证接口

`POST /api/auth/login`

输入：

- `username`
- `password`

- `access_token`
- `token_type`
- `user`

`POST /api/chat/ask`

输入：

- `doc_type`

- `conversation_id`
- `answer`
- `grounded`
- `sources`

`POST /api/knowledge/import`

输入：

- 文件
- `book_title`
- `doc_type`
- `allowed_role`

- `job_id`
- `status`
- `message`

`GET /api/conversations`

输出：

- 当前用户的历史列表

## 8. 前端方案

### 8.1 设计目标

- 足够轻量
- 能演示完整闭环
- 不引入过多状态管理复杂度

- 登录页

- 历史记录页

### 8.3 登录态方案

当前采用：

- token 存本地
- 用户信息存本地
- 页面级守卫控制访问

设计原因：

- 前后端分离场景下实现快
- 首版易于理解

后续可以升级为：

- HttpOnly Cookie
- 刷新 token
- 更细粒度权限路由控制

## 11. 安全与风险

### 11.1 当前风险

- 如果 `OPENAI_API_KEY` 未配置，问答会失败
- 默认管理员密码过于简单，只适合本地开发
- 当前登录态使用 localStorage，不适合高安全要求场景
- SQLite 与 FAISS 更适合单机开发，不适合复杂生产环境

### 11.2 当前应对策略

- README 中明确要求修改 `JWT_SECRET_KEY`
- 角色隔离至少落实到导入权限与检索过滤
- 回答策略偏保守，降低无依据生成风险

### 11.3 后续优化方向

- 更安全的会话方案
- PostgreSQL 替代 SQLite
- 云向量库替代 FAISS
- 更细粒度文档权限控制

## 13. 方案总结

本方案的核心思路是：

- 先做一个真正能跑的出版知识助手
- 用 LangChain 打好智能能力层基础
- 用分层结构控制复杂度
- 用 RAG 作为首版主能力
- 为后续多模态、多 Agent 和更多出版业务能力留足接口

这也是为什么 V1 更强调“工程可落地”而不是“功能一次做全”。

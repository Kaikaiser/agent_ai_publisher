# RAG 专项设计文档

## 1. 这份文档解决什么问题

这份文档专门解释本项目中的 RAG 设计，包括：

- 为什么这个项目要使用 RAG
- 当前 RAG 的整体流程是什么
- 文档是怎么导入、切片、向量化和检索的
- 问答时如何把检索结果交给模型
- 当前方案的优点、限制和后续优化方向

## 2. 为什么这个项目要用 RAG

这个项目是“出版社 AI 助手”，核心场景不是闲聊，而是围绕图书、教材、教辅、资料内容进行回答。

如果只依赖大模型本身，会有几个问题：

- 模型不知道你的私有资料
- 容易脱离文档内容自由发挥
- 回答缺乏出处
- 很难做按书名、文档类型、权限过滤

因此需要引入 RAG。

RAG 的核心思想是：

- 先检索
- 再生成

也就是：

1. 用户提问
2. 系统先从知识库里检索相关内容
3. 把检索结果作为上下文交给模型
4. 模型基于这些上下文生成回答

## 3. 当前 RAG 在项目中的定位

在本项目里，RAG 不是辅助功能，而是 V1 的主能力之一。

它承担的职责是：

- 让问答基于出版社资料进行
- 提高答案可信度
- 返回来源引用
- 为后续知识管理和检索优化打基础

## 4. 当前 RAG 总体流程

当前 RAG 链路可以分为两段：

### 4.1 离线链路

也就是知识导入链路：

1. 上传文件
2. 解析文本
3. 切片
4. 向量化
5. 写入 FAISS 索引
6. 保存文档元数据

### 4.2 在线链路

也就是问答链路：

1. 用户提问
2. 检索相似片段
3. 按角色与过滤条件筛选
4. 拼接上下文
5. 调用模型生成答案
6. 返回答案和来源

## 5. 离线链路设计

### 5.1 数据来源

当前支持的知识来源是本地文件导入。

支持类型：

- TXT
- Markdown
- PDF
- DOCX

设计原因：

- 本地文件最容易启动
- 最适合本地开发和演示
- 后续容易扩展为 CMS 或数据库同步

### 5.2 文件解析

代码位置：

- `backend/app/knowledge/loaders/file_loader.py`

不同文件类型采用不同解析方式：

- TXT / Markdown
  - 直接读取全文
- PDF
  - 逐页提取文本
- DOCX
  - 读取段落内容并拼接

输出统一转换成 LangChain `Document` 对象。

这样做的好处是：

- 上游文件格式不同
- 下游切片和检索逻辑却可以统一处理

### 5.3 元数据设计

导入时，会为文档附加元数据，比如：

- `filename`
- `book_title`
- `doc_type`
- `allowed_role`
- `location`
- `document_id`

这些元数据非常重要，因为后续不仅要做相似度搜索，还要做：

- 权限过滤
- 书名过滤
- 文档类型过滤
- 来源展示

也就是说，这个项目的 RAG 不是“只存文本”，而是“文本 + 元数据”一起设计的。

### 5.4 切片策略

代码位置：

- `backend/app/knowledge/splitter.py`

当前使用：

- `RecursiveCharacterTextSplitter`

当前参数：

- `chunk_size = 700`
- `chunk_overlap = 100`

### 5.5 为什么这样切

切片的目标是平衡两件事：

- 片段不能太大，否则检索不准
- 片段不能太小，否则上下文不完整

当前选择 700 / 100 的理由：

- 适合大多数教材、教辅、说明性文本
- 对中文场景相对稳
- 首版实现简单、可调

### 5.6 向量化

代码位置：

- `backend/app/services/providers.py`

当前向量模型由：

- `OpenAIEmbeddings`

提供。

也就是说，导入文档后，每个切片都会被转换成向量表示。

### 5.7 向量存储

代码位置：

- `backend/app/knowledge/vector_store.py`

当前使用：

- FAISS 本地向量库

当前逻辑：

- 如果索引已存在，则追加文档
- 如果索引不存在，则创建新索引

保存路径：

- `backend/data/vector_store/faiss_index`

这样设计适合本地开发、单机调试和首版验证。

## 6. 在线检索链路设计

### 6.1 入口在哪里

问答请求进入：

- `backend/app/services/chat.py`

`ChatService.ask()` 会先定义一个 `search_func`，供后续编排层调用。

### 6.2 检索是怎么做的

实际调用：

- `VectorStoreService.search()`

流程如下：

1. 读取本地 FAISS 索引
2. 对用户问题做向量化
3. 执行相似度搜索
4. 对结果做元数据过滤

### 6.3 当前过滤条件

当前支持：

- 角色过滤：`allowed_role`
- 书名过滤：`book_title`
- 文档类型过滤：`doc_type`

### 6.4 为什么角色过滤很重要

因为这个项目有权限概念。

如果一个文档只允许管理员访问，那么普通用户检索时不应该命中它。

这也是为什么文档元数据里必须保存 `allowed_role`。

### 6.5 当前召回策略

当前策略比较朴素：

- 相似度 Top-K 检索
- 不做 rerank
- 不做混合检索

这是 V1 的有意取舍。

优点：

- 简单稳定
- 容易理解
- 调试成本低

缺点：

- 复杂问题时召回质量不一定最优
- 对关键词和结构信号的利用还不够强

## 7. 生成链路设计

### 7.1 谁负责把检索结果交给模型

代码位置：

- `backend/app/agents/orchestration/service.py`

当前由：

- `AgentOrchestrator`

负责。

### 7.2 生成流程

当检索到文档后：

1. 把每个文档片段格式化为上下文文本
2. 拼上来源信息：
   - 文件名
   - 书名
   - 文档类型
   - 位置
3. 构造 Prompt
4. 调用 LLM `invoke()`
5. 生成最终回答

所以 RAG 的关键不是“检索完直接返回”，而是“把检索结果组织成适合模型理解的上下文”。

### 7.3 没检索到怎么办

当前策略是保守回答：

- 直接返回 `知识库中未找到充分依据。`

这样设计的原因：

- 出版场景更强调正确性
- 比起“猜一个答案”，明确说找不到依据更安全

## 8. 来源引用设计

### 8.1 为什么要返回来源

在出版社场景里，用户往往关心：

- 你这个回答依据哪本资料
- 是来自哪个文件
- 是不是能追溯到原文

所以当前回答不仅返回 `answer`，还会返回 `sources`。

### 8.2 来源结构

每条来源当前包含：

- `content`
- `filename`
- `book_title`
- `doc_type`
- `location`

这让前端能够把回答和引用来源一起展示出来。

## 9. 当前 RAG 方案的优点

### 9.1 对业务足够贴合

它不是通用聊天增强，而是针对出版社资料问答设计的。

### 9.2 工程结构清晰

导入、切片、向量化、检索、生成都拆分在独立模块里。

### 9.3 易于本地运行

SQLite + FAISS 很适合本地开发，不需要一开始就依赖云服务。

### 9.4 适合逐步增强

当前架构可以自然演进到：

- rerank
- 混合检索
- 更细元数据过滤
- 更复杂的 Agent 编排

## 10. 当前 RAG 方案的限制

### 10.1 没有 rerank

当前相似度召回后直接用，不做二次排序。

### 10.2 没有混合检索

当前只做向量检索，没有结合关键词或 BM25。

### 10.3 元数据还比较少

当前主要是：

- 书名
- 文档类型
- 角色

还没有做到：

- 学科
- 年级
- 册次
- 章节

### 10.4 PDF 质量受原文影响

如果 PDF 本身抽取质量差，RAG 效果会受影响。

## 11. 推荐的 RAG 优化路线

### 11.1 第一阶段优化

优先级最高的建议：

1. 增加更细元数据
- 学科
- 年级
- 册次
- 章节

2. 增加检索调试接口
- 查看命中的 chunk
- 查看过滤条件
- 查看召回结果排序

3. 优化来源展示
- 更清晰地展示命中位置

### 11.2 第二阶段优化

1. 引入 rerank
2. 引入混合检索
3. 为不同文档类型设计不同切片策略
4. 增加文档重建索引能力

### 11.3 第三阶段优化

1. 接入 OCR 文档链路
2. 多模态检索
3. 多知识库路由
4. 多 Agent 协作检索与回答

## 12. 当前 RAG 的完整链路总结

可以把当前 RAG 概括成一句话：

- 管理员把出版社资料导入系统
- 系统把资料切片并向量化存储
- 用户提问时，系统先检索相关资料
- 再把检索结果交给模型生成答案
- 最后把答案和来源一起返回

也就是：

```text
文档导入 -> 切片 -> 向量化 -> FAISS存储
用户提问 -> 相似度检索 -> 元数据过滤 -> 模型生成 -> 返回答案与来源
```

## 13. 建议你阅读 RAG 代码的顺序

如果你准备看这一块代码，推荐顺序是：

1. `backend/app/knowledge/loaders/file_loader.py`
2. `backend/app/knowledge/splitter.py`
3. `backend/app/knowledge/vector_store.py`
4. `backend/app/services/knowledge.py`
5. `backend/app/services/chat.py`
6. `backend/app/agents/orchestration/service.py`

这样你会比较容易看懂：

- 文档是怎么进来的
- 文本是怎么被切开的
- 向量是怎么存进去的
- 提问时又是怎么被取出来的

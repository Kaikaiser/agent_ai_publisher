# 拍照搜题功能设计文档

## 1. 功能目标

本次扩展为“出版社 AI 助手”增加了拍照搜题第一版能力。

当前版本的目标不是一步做到完整视觉解题系统，而是先把图片输入接入现有问答架构，形成一条可运行、可理解、可迭代的链路。

当前能力边界：

- 用户上传题目图片
- 系统先对图片做 OCR 文本识别
- 用户可以查看并修改识别出的题目文本
- 用户确认后，系统复用现有问答链路进行回答

## 2. 为什么采用“两步式”设计

当前没有直接把图片上传后立刻自动解题，而是采用两步式流程：

1. 先识别题目
2. 再确认题目并问答

这样设计的原因：

- OCR 识别难免有误差
- 题目文本错误会直接影响后续回答质量
- 让用户在问答前有一次人工确认，可以显著降低错误传播
- 比“一步式自动解题”更适合作为第一版稳定落地

## 3. 功能流程

### 3.1 用户侧流程

在前端“拍照搜题”区域，用户操作顺序是：

1. 选择题目图片
2. 点击“识别题目”
3. 查看识别结果
4. 如有需要，手动修改识别文本
5. 点击“确认题目并问答”
6. 查看答案与引用来源

### 3.2 后端流程

完整链路如下：

1. 前端上传图片到 `/api/chat/recognize-image`
2. 后端调用 OCR 服务提取题目文本
3. 前端展示 OCR 结果给用户
4. 用户确认后，前端把文本提交给 `/api/chat/ask`
5. 后端复用当前问答主链路：
   - 检索知识库
   - 组织 Prompt
   - 调用模型回答
   - 保存历史记录
6. 返回答案、来源和对话记录 ID

## 4. 后端设计

### 4.1 新增接口

当前新增了一个识别接口：

- `POST /api/chat/recognize-image`

已有接口继续复用：

- `POST /api/chat/ask`
- `POST /api/chat/ask-image`

其中：

- `recognize-image` 用于两步式流程的第一步
- `ask-image` 保留为一步式图片问答入口
- `ask` 继续负责纯文本问答

### 4.2 核心代码位置

主要涉及这些文件：

- `backend/app/api/routes/chat.py`
- `backend/app/services/chat.py`
- `backend/app/services/image_ocr.py`
- `backend/app/schemas/chat.py`

### 4.3 OCR 服务

OCR 能力封装在：

- `ImageOCRService`

它当前的职责是：

- 接收图片字节流
- 把图片转为 base64
- 调用多模态模型提取题目文本
- 返回识别结果

提示词策略是：

- 只提取题目、公式和选项
- 不解题
- 不补充说明
- 尽量保持原始结构

这样做是为了把 OCR 识别和问答生成分开，职责更清晰。

### 4.4 问答服务扩展

`ChatService` 新增了：

- `recognize_image()`
- `ask_from_image()`

职责分工如下：

- `recognize_image()`
  - 只负责 OCR
- `ask_from_image()`
  - 先 OCR，再直接问答
- `ask()`
  - 纯文本问答

当前两步式流程主要走：

- `recognize_image()`
- 再由前端调用 `ask()`

## 5. 前端设计

### 5.1 页面改动

改动页面：

- `frontend/src/pages/ChatPage.jsx`

当前在问答页新增了一个“拍照搜题”区域。

### 5.2 页面状态

新增状态主要有：

- `imageFile`
- `recognizedText`

其中：

- `imageFile` 保存用户选中的图片
- `recognizedText` 保存 OCR 识别出的文本，并允许用户编辑

### 5.3 前端交互逻辑

拍照搜题当前拆成两个动作：

#### 动作 1：识别题目

前端调用：

- `POST /api/chat/recognize-image`

返回：

- `recognized_text`

然后把识别结果填进文本框。

#### 动作 2：确认题目并问答

前端调用：

- `POST /api/chat/ask`

请求体中把 `recognizedText` 作为 `question` 提交。

这样可以完全复用已有文本问答逻辑，而不需要再造一套独立问答通道。

## 6. 与现有 RAG 的关系

拍照搜题当前并不是单独的新问答系统，而是“图片输入前置 + 复用现有 RAG”。

换句话说：

- 图片负责把题目带进系统
- RAG 负责继续做检索增强问答

这意味着拍照搜题当前的回答质量，仍然受这些因素影响：

- OCR 识别质量
- 检索召回质量
- 知识库内容是否覆盖该题目相关知识
- 模型回答能力

## 7. 当前接口设计

### 7.1 识别图片接口

- 方法：`POST`
- 路径：`/api/chat/recognize-image`
- 请求类型：`multipart/form-data`
- 必填字段：`file`

返回：

```json
{
  "recognized_text": "已识别题目：1+1等于几？"
}
```

### 7.2 图片直接问答接口

- 方法：`POST`
- 路径：`/api/chat/ask-image`
- 请求类型：`multipart/form-data`
- 说明：保留为一步式接口

### 7.3 文本问答接口

- 方法：`POST`
- 路径：`/api/chat/ask`
- 说明：两步式流程最终仍然走这个接口

- `5 passed`

## 9. 当前限制

当前版本仍有这些限制：

- OCR 依赖多模态模型能力
- 还没有图片预览
- 还没有题型判断
- 还没有专门的解题步骤模板
- 还没有针对数学公式做更强的结构化识别

## 10. 下一步建议

拍照搜题后续推荐按这个顺序继续增强：

1. 图片预览
2. 题型判断
3. 更完整的解题步骤输出
4. 数学公式识别优化
5. 拍照搜题专属 Prompt
6. 题目解析与知识点提取

## 11. 设计总结

这次扩展的核心思想是：

- 不新造一整套问答系统
- 先把图片输入接进现有架构
- 用“两步式确认”降低 OCR 错误影响
- 继续复用当前 RAG 和问答主链路

这保证了第一版拍照搜题既能落地，又不会破坏现有系统结构。

# 二次开发指南

## 1. 这份文档适合谁

这份文档面向：

- 继续维护这个项目的开发者
- 想新增功能的初学者
- 想基于当前工程继续扩展的团队成员

它主要解决的问题是：

- 新功能应该改哪里
- 新模块怎么接入当前结构
- 哪些地方可以扩展，哪些地方暂时不要乱动

## 2. 开发前先理解什么

在动手改代码之前，建议先看这几份文档：

1. `README.md`
2. `docs/ARCHITECTURE_DESIGN.md`
3. `docs/BACKEND_RUNTIME_FLOW.md`
4. `docs/FRONTEND_RUNTIME_FLOW.md`
5. `docs/RAG_DESIGN.md`

这样你会先理解：

- 项目目标
- 目录职责
- 后端主链路
- 前端主链路
- RAG 主链路

## 3. 当前项目的扩展原则

二次开发时，优先遵守这几个原则：

### 3.1 不要把业务逻辑直接写进路由

正确方式：

- 路由层只负责接收请求和返回响应
- 业务逻辑尽量放进 `services`

### 3.2 不要在页面里堆太多请求逻辑

正确方式：

- API 调用统一走 `frontend/src/api/client.js`
- 页面层只负责交互和展示

### 3.3 不要把模型提供方细节散落到业务层

正确方式：

- 模型和 embedding 的创建统一经由 `providers.py`

### 3.4 不要把检索逻辑和页面逻辑耦合

正确方式：

- 检索逻辑只在后端 `knowledge`、`services`、`agents` 层演进

## 4. 如果你要新增后端接口，建议怎么做

推荐顺序：

1. 先定义数据结构
- 看是否需要新增 schema
- 看是否需要新增数据库字段

2. 再写服务层
- 先把业务逻辑放进 `services`

3. 最后接入 API 路由
- 在 `api/routes` 里增加接口

例如你要新增“管理员创建普通用户”：

- 在 `auth/service.py` 加创建用户逻辑
- 在 `schemas` 增加请求响应模型
- 在 `api/routes` 增加接口
- 前端增加页面或表单

## 5. 如果你要新增前端页面，建议怎么做

推荐顺序：

1. 明确这个页面对应哪个后端接口
2. 在 `src/pages` 新建页面文件
3. 在 `App.jsx` 里加路由
4. 如果要控制角色访问，把角色判断放在布局或路由层
5. 样式先复用现有 CSS，不要一开始就大改视觉系统

## 6. 如果你要新增 RAG 能力，应该改哪里

### 6.1 新增文档解析器

改：

- `backend/app/knowledge/loaders/file_loader.py`

### 6.2 调整切片策略

改：

- `backend/app/knowledge/splitter.py`

### 6.3 调整检索逻辑

改：

- `backend/app/knowledge/vector_store.py`

### 6.4 调整问答编排

改：

- `backend/app/agents/orchestration/service.py`
- `backend/app/services/chat.py`

### 6.5 新增过滤维度

通常要一起改：

- 文档元数据写入逻辑
- 检索过滤逻辑
- 前端筛选表单
- API 请求参数

## 7. 如果你要新增模型提供方，应该怎么做

当前模型创建入口在：

- `backend/app/services/providers.py`

建议做法：

1. 新建 provider 类
2. 保持和 `OpenAIProvider` 类似的接口
3. 统一返回：
   - `get_llm()`
   - `get_embeddings()`
4. 再通过配置切换

这样以后可以平滑支持：

- Ollama
- Azure OpenAI
- 兼容 OpenAI API 的平台

## 8. 如果你要升级成更强的 Agent，应该怎么做

当前 Agent 编排入口在：

- `backend/app/agents/orchestration/service.py`

建议演进路径：

1. 先增强单 Agent 工具调用
2. 再增加 router / planner 抽象
3. 最后再做多 Agent 协作

不要一开始就直接堆复杂多 Agent，否则：

- 调试成本很高
- 出错范围很大
- 对当前业务闭环提升不明显

## 9. 当前最适合继续做的功能

推荐优先级：

### 9.1 第一优先级

- 管理员创建普通用户
- 更细错误提示
- 优化前端体验
- 增强 README 和使用流程

### 9.2 第二优先级

- 增加学科、年级、册次等元数据
- 增加检索调试接口
- 增加 rerank
- 增加更多测试

### 9.3 第三优先级

- OCR 接入
- 图片输入
- 语音识别
- 多 Agent 编排

## 10. 不推荐你现在立刻做的事

以下事情目前不建议优先做：

- 一上来重构成复杂微服务
- 一上来换成很多框架
- 一上来做完整 RBAC
- 一上来做复杂多 Agent 系统
- 一上来做过度抽象

原因：

- 当前项目还在建立稳定主链路阶段
- 现在最重要的是让结构稳、功能通、扩展明确

## 11. 开发时的自检清单

每次加新功能，建议都问自己：

1. 这个功能改动是在正确层吗？
2. 业务逻辑是不是写进了 service，而不是 route？
3. 前端是不是复用了统一 API 请求方法？
4. 有没有影响已有登录、导入、问答、历史主链路？
5. 有没有补测试或至少补说明文档？

## 12. 测试建议

当前测试集中在：

- 登录
- API 主流程
- 计算器工具

如果你新增功能，建议至少补其中一种：

- 单元测试
- API 集成测试
- 最小联调说明

## 13. 推荐开发顺序

如果你准备继续推进这个项目，建议按这个顺序迭代：

1. 用户管理
2. 错误处理优化
3. 检索过滤增强
4. RAG 质量提升
5. 前端体验提升
6. OCR / 图片输入
7. 语音输入
8. 多 Agent

## 14. 一句话总结

这个项目当前最正确的二次开发方式不是“大改架构”，而是：

- 保持分层
- 沿着现有链路补能力
- 先做高价值、低风险、可验证的增强

这样项目会越来越稳，而不是越改越乱。

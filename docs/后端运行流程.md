# 后端运行流程

本文档按“服务实际怎么跑”的顺序说明当前后端逻辑，重点覆盖登录、知识导入、RAG 问答、拍照搜题，以及模型提供方切换。

## 1. 启动入口

入口文件：`backend/app/main.py`

启动时主要完成这些事：

1. 创建 FastAPI 应用
2. 配置 CORS
3. 注册路由
4. 初始化数据库
5. 自动确保默认管理员存在

因此，后端启动后即具备：

- 登录能力
- 管理员知识导入能力
- 问答能力
- 历史记录查询能力

## 2. 配置加载

配置入口：`backend/app/core/config.py`

配置来源：仓库根目录 `.env`

关键点：

- 使用 `pydantic-settings`
- 支持 `openai / ark / glm` 三种提供方
- 文本模型、视觉模型、向量模型分别配置
- 启动后所有服务共享同一份配置对象

## 3. Provider 选择流程

Provider 实现：`backend/app/services/providers.py`

业务层不会直接关心“当前用的是哪家模型”，而是通过统一接口获取：

- `get_llm()`
- `get_vision_llm()`
- `get_embeddings()`

### 3.1 文本问答模型

- `openai` -> `OPENAI_MODEL`
- `ark` -> `ARK_CHAT_MODEL`
- `glm` -> `GLM_CHAT_MODEL`

### 3.2 图片 OCR 模型

- 优先使用独立视觉模型
- 若未单独配置，则回退到文本模型

例如 GLM：

- 优先 `GLM_VISION_MODEL`
- 否则回退到 `GLM_CHAT_MODEL`

### 3.3 向量模型

- `openai` -> `OPENAI_EMBEDDING_MODEL`
- `ark` -> `ARK_EMBEDDING_MODEL`
- `glm` -> `GLM_EMBEDDING_MODEL`

## 4. 鉴权流程

相关文件：

- `backend/app/api/routes/auth.py`
- `backend/app/auth/service.py`
- `backend/app/auth/dependencies.py`

流程如下：

1. 前端向 `/api/auth/login` 提交用户名和密码
2. `AuthService` 校验账号、密码和用户状态
3. 校验成功后签发 JWT
4. 前端保存 token
5. 后续请求通过 `Authorization: Bearer <token>` 访问受保护接口

管理员接口额外通过 `require_admin()` 校验角色。

## 5. 知识导入流程

相关文件：

- `backend/app/api/routes/knowledge.py`
- `backend/app/services/knowledge.py`
- `backend/app/knowledge/loaders/file_loader.py`
- `backend/app/knowledge/splitter.py`
- `backend/app/knowledge/vector_store.py`

完整流程：

1. 管理员上传文件到 `/api/knowledge/import`
2. 文件先保存到上传目录
3. `KnowledgeService.import_file()` 创建导入任务记录
4. 按文件类型解析内容
5. 切片
6. 用当前 provider 的 embedding 模型做向量化
7. 写入本地 FAISS 索引
8. 导入任务状态改为 `completed`

### 本次调试补充

TXT / Markdown 读取使用 `utf-8-sig`，用于自动去掉 BOM。

这能避免：

- 知识内容前面出现不可见脏字符
- 终端调试输出异常
- 生成回答时引用片段被 BOM 污染

## 6. 文本问答流程

相关文件：

- `backend/app/api/routes/chat.py`
- `backend/app/services/chat.py`
- `backend/app/agents/orchestration/service.py`

流程如下：

1. 前端调用 `/api/chat/ask`
2. 路由层注入：
   - 当前用户
   - 文本 LLM
   - embeddings
3. `ChatService.ask()` 进入主问答流程
4. `VectorStoreService.search()` 先做知识检索
5. `AgentOrchestrator.run()` 根据检索结果决定回答方式
6. 回答结果和来源写入 `ConversationRecord`
7. 返回：
   - `answer`
   - `grounded`
   - `sources`
   - `conversation_id`

### 当前判定逻辑

- 如果没有检索结果，则返回“知识库中未找到充分依据”
- 如果有检索结果，则把上下文拼接进 prompt 再交给模型回答
- 如果问题是纯数学表达式且没有文档上下文，则直接调用计算器工具

## 7. 拍照搜题流程

相关文件：

- `backend/app/api/routes/chat.py`
- `backend/app/services/image_ocr.py`
- `backend/app/services/chat.py`

有两条接口：

### 7.1 仅识别图片文本

- `POST /api/chat/recognize-image`

流程：

1. 校验上传文件是否为图片
2. 注入 `vision_llm`
3. `ImageOCRService.extract_text()` 抽取题目文本
4. 返回 `recognized_text`

### 7.2 识别后直接问答

- `POST /api/chat/ask-image`

流程：

1. 先做 OCR
2. 用识别结果作为问题文本
3. 继续复用现有 `/ask` 的检索与问答逻辑
4. 返回问答结果和 `recognized_text`

## 8. 历史记录流程

相关文件：

- `backend/app/api/routes/conversations.py`
- `backend/app/services/chat.py`
- `backend/app/db/models.py`

每次问答完成后都会：

1. 保存问题
2. 保存答案
3. 保存是否 grounded
4. 保存来源摘要 JSON

前端历史页通过 `/api/conversations` 查询当前用户记录。

## 9. 本次 GLM 联调结果

本次已完成真实调试，不是只靠单元测试。

### 已验证通过

- 登录
- GLM 向量化导入
- FAISS 检索
- GLM 文本问答

### 实际现象

- 示例教材可成功导入
- 检索可召回相关片段
- 对“这本教辅适合几年级？”这类问题，系统可正确回答“三年级”

### 遇到的问题与处理

#### 问题 1：后端进程未读取最新 `.env`

处理：

- 清理残留 uvicorn 进程
- 使用单进程方式重启后端
- 避免 `--reload` 在 Windows 下生成多进程导致排障复杂化

#### 问题 2：PowerShell 中文请求体与输出编码干扰调试

处理：

- 关键联调改用 UTF-8 Python 脚本调用本地接口
- 终端仅作为辅助，不再依赖 PowerShell 直接观察中文业务结果

#### 问题 3：示例 TXT 带 BOM

处理：

- 文本加载器改用 `utf-8-sig`
